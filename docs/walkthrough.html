What is Coxswain, really?

You have a repo and a pile of small tasks. You could open a coding agent and babysit it. Coxswain is the layer that lets you say “work this issue” to one assistant and get back a reviewed pull request — without you watching, and without burning tokens on plumbing.

The crew metaphor → A coxswain is the one rower who doesn’t row. They sit at the stern, steer, and call the strokes. That’s the whole design in one image: you talk to the cox; the crew (worker agents) rows; nothing ships unreviewed. Every metaphor below is followed by the real component it maps to — the metaphor is training wheels, not the machine.

Why this exists

It’s the fix for a real post-mortem

Coxswain is the successor to a project called relay. Relay worked, but it burned tokens on small tasks and felt fragile. Every design choice here is a named fix. If you only remember three:

🔁 No cold restarts

Relay spun up a fresh agent for every phase — each one re-read the whole repo. Coxswain keeps one session per task and resumes it for fixes, so a fix costs the delta, not a re-read.

🚫 No retry loops on the pricey model

Relay re-ran its reviewer up to ~5× when infra flaked. Here, every agent call has retry cap 1 — a flake pings you, it never silently re-runs.

💤 Zero-token waiting

Watching for “is it done yet?” spends no tokens: a plain Python watcher classifies events and only wakes the smart layer when something actually needs a human.
